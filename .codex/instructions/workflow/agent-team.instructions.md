# Agent Team Instructions

## 目标

Agent team 用于把噪音大的探索、审查、日志分析和验收检查移出主会话。主会话只保留需求、决策、结论、验证结果和下一步，避免上下文污染和频繁压缩。

## 默认职责

主 agent：

1. 确定范围、计划、风险和验收标准。
2. 决定是否启用 subagent。
3. 执行或协调最终文件修改。
4. 合并 subagent 结论并做取舍。
5. 向用户交付简明结果。

Subagent：

1. 默认只读。
2. 只处理明确、边界有限的任务。
3. 返回摘要，不返回完整探索过程。
4. 不提交、不 stage、不移动归档、不改版本号。

## 推荐角色

| 角色 | 使用时机 | 输出 |
| --- | --- | --- |
| Architecture Reviewer | Rust/Tauri 分层、domain、provider、jobs、SQLite 变更 | 分层风险、阻塞问题、建议 |
| Frontend UX Reviewer | React、TypeScript、视觉、交互状态变更 | UI 状态缺口、响应式风险、文案问题 |
| Privacy/Security Reviewer | API Key、音频、转写文本、日志、云端调用 | 泄露风险、边界遗漏、脱敏建议 |
| Test/Verification Reviewer | 测试失败、验证矩阵、构建输出 | 根因摘要、缺口、最小验证建议 |
| Release/Docs Reviewer | 版本完成、发布说明、归档、Git 流程 | 一致性问题、缺失文档、发布阻塞 |

## 何时启用

建议启用：

1. 跨 Rust / TypeScript / SQLite 契约。
2. 修改 provider、模型调用、隐私边界。
3. 删除旧路径或做架构迁移。
4. 版本验收、功能 review、发布前检查。
5. 测试日志很长，主会话只需要结论。

不建议启用：

1. 单文件低风险文档修正。
2. 错别字、链接、注释。
3. 已有测试覆盖且无行为变化的小改动。
4. 用户要求只由主 agent 处理。

## 并行 / 串行

并行适合读多写少：

1. 各自审查不同层。
2. 一个查旧路径残留，一个查测试缺口，一个查文档一致性。
3. 多份长日志或长文档摘要。

串行适合写操作和依赖链：

1. 多个任务会触碰同一文件。
2. 数据库 schema、版本号、发布归档。
3. 一个结论会改变下一个任务范围。
4. 需要用户先确认方案。

## Prompt 模板

```text
你是 [角色]。只读审查以下范围：[文件/目录/diff]。
依据：[相关 instruction 或 AI文档]。
只输出：
1. 阻塞问题
2. 建议修复
3. 测试/验证缺口
4. 一句话结论
不要修改文件，不要输出大段源码、日志、密钥、完整用户文本或本地隐私路径。
```

## 输出限制

1. 默认摘要不超过 20 行。
2. 每个问题必须有文件路径或明确证据。
3. 大段日志只摘关键错误行。
4. 不确定时标记“需主 agent 复核”，不要扩大范围。

## 冲突处理

主 agent 按以下顺序取舍：

1. 安全、隐私、数据损坏。
2. 构建、测试、运行阻塞。
3. 当前版本验收标准。
4. 架构边界。
5. 视觉、文案、开发体验。

只有产品取舍、破坏性操作、范围扩大或无法判断的业务决策需要问用户。

## Handoff 摘要

长任务或接近上下文压缩时，主 agent 维护短 handoff：

```text
已完成：
当前变更：
验证结果：
未解决风险：
下一步：
禁止触碰：
```

Handoff 只记录可继续工作的事实，不记录完整探索过程。
