# Review / Subagent Workflow Instructions

## 适用场景

本文件用于约束 code review、功能 review 和 subagent 协作。它是工作流规则，不替代具体代码规范。

## Code Review

完成一个明确 plan 或较大代码切片后，可进行 code review。重点检查：

1. 是否符合当前版本目标。
2. 是否破坏分层边界。
3. 是否重新引入旧 Qwen / llama.cpp Todo runtime。
4. DTO / TypeScript 类型 / SQLite schema 是否一致。
5. 错误处理、日志、隐私边界是否合规。
6. 是否有必要测试，验证命令是否覆盖风险。

输出应优先列问题：

```text
阻塞问题
建议修复
测试缺口
结论
```

没有问题时明确写“未发现阻塞问题”，并说明剩余风险。

## 功能 Review

每个版本完成开发后进行功能 review。重点检查：

1. 是否满足 `AI文档/03-版本迭代/` 中本版本验收标准。
2. 是否有可操作的用户路径。
3. 前端是否体现录音、转写、语义、隐私、失败状态。
4. 文档、版本号、发布说明、归档记录是否一致。
5. 最小验证是否通过。

## Subagent 使用边界

使用 subagent 时应给出明确、只读或限定范围的任务：

1. “审查这些文件是否违反分层边界。”
2. “检查前端 UI 是否存在明显状态缺口。”
3. “对照版本验收标准做功能 review。”
4. “检查是否残留 Qwen / llama.cpp Todo 路径。”

默认策略：

1. Subagent 默认只读、只输出报告。
2. Subagent 不直接提交、不 stage、不改版本号、不移动归档目录。
3. 如需让 subagent 编辑文件，必须明确文件范围、禁止触碰范围和验证命令。
4. 主 agent 负责合并结论、取舍建议、执行最终修改和交付说明。

避免：

1. 让 subagent 自行决定大范围重构。
2. 让多个 subagent 同时编辑同一批文件。
3. 让 subagent 读取或输出密钥、完整用户文稿、完整本地音频路径。
4. 为很小的单文件改动强制多轮 review，造成过度流程。

## Agent Team 角色

复杂版本开发可以按需组成临时 agent team：

| 角色 | 适用任务 | 默认权限 |
| --- | --- | --- |
| Architecture Reviewer | Rust/Tauri 分层、provider、jobs、SQLite schema | 只读报告 |
| Frontend UX Reviewer | React 结构、视觉、状态、桌面交互 | 只读报告 |
| Privacy/Security Reviewer | API Key、音频、转写文本、日志脱敏、云端调用边界 | 只读报告 |
| Test/Verification Reviewer | 测试缺口、验证命令、失败日志摘要 | 只读报告 |
| Release/Docs Reviewer | Git、版本号、发布说明、归档记录一致性 | 只读报告 |

角色不是固定编制。只启用能降低风险或节省主会话上下文的角色。

## 并行与串行

可以并行：

1. 独立目录或独立主题的只读审查。
2. 日志、测试输出、文档验收、UI 截图问题归类。
3. “是否残留旧路径”这类可独立搜索的检查。

必须串行：

1. 多个 agent 会编辑同一文件或相邻模块。
2. SQLite schema、迁移、版本号、发布归档。
3. 一个 agent 的结论会改变另一个 agent 的检查范围。
4. 用户明确要求主 agent 先确认方案。

## 结论合并

主 agent 汇总多 agent 输出时使用以下优先级：

1. 安全、隐私、数据损坏风险。
2. 构建、测试、运行阻塞。
3. 当前版本验收标准。
4. 架构边界和长期维护成本。
5. 视觉、文案、开发体验建议。

冲突意见必须由主 agent 决策，不把分歧原样丢给用户。只有产品取舍、破坏性变更、范围扩大时才询问用户。

## 上下文控制

为避免主会话频繁压缩：

1. Subagent 返回不超过 20 行摘要，除非发现阻塞问题。
2. 摘要只包含结论、文件位置、证据、建议，不粘贴大段日志或源码。
3. 大输出保留在命令结果、临时文件或子线程中，主会话只引用关键行。
4. 每个开发阶段结束时，主 agent 维护短状态摘要：已完成、未完成、验证、风险、下一步。
5. 若上下文接近拥挤，优先生成 `handoff` 摘要，而不是继续追加探索细节。

## 过度 Review 判断

以下情况通常不需要 subagent review：

1. 只改错别字、链接、注释。
2. 只新增无逻辑的文档索引。
3. 已有测试覆盖且改动低风险、单文件、无行为变化。

以下情况建议 review：

1. 跨 Rust / TypeScript 契约。
2. 修改 SQLite schema 或任务状态机。
3. 修改 provider、模型调用、隐私边界。
4. 版本验收前。
5. 修复高风险 bug 或删除旧路径。
