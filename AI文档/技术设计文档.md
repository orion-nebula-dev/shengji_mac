# 技术设计文档

## 1. 文档目标

本文档用于描述智能 Todo 一期的系统架构、模块设计、数据流、状态机、任务流与关键实现建议，作为研发落地依据。

## 2. 设计目标

1. 跑通“持续录音 -> 分段转写 -> 空闲触发聚合 -> Todo 提取 -> Todo 管理”闭环。
2. 支持在设置中配置切片时长、空闲触发时间以及双模型参数。
3. 实现转写模型与 Todo 提取模型解耦。
4. 保证数据可追踪、可回溯、可排障。
5. 支持应用内嵌本地 Todo 模型，并保留云端 Provider 兜底能力。

## 3. 总体架构

### 3.1 架构分层

| 层级 | 说明 |
| --- | --- |
| UI 展示层 | 设置页、Todo 列表、Todo 详情、运行状态面板 |
| 应用编排层 | 录音控制、转写调度、会话聚合调度、Todo 提取调度 |
| 领域服务层 | SettingsService、RecordingService、TranscriptionService、ConversationService、TodoService |
| 基础设施层 | SQLite、音频采集能力、Keychain、安全存储、网络客户端 |
| Provider 抽象层 | ASR Provider、Todo Extraction Provider |
| 本地模型运行时层 | ModelRuntimeManager、模型目录管理、子进程生命周期管理、健康检查 |

### 3.2 模块图

```text
设置与配置模块
  -> RecordingService
  -> ProviderConfigService

音频采集模块
  -> audio_segments
  -> TranscriptionService

语音转写模块
  -> transcript_segments
  -> ConversationService

会话聚合模块
  -> conversation_sessions
  -> TodoExtractionService

Todo 提取模块
  -> todos
  -> processing_jobs
```

### 3.3 当前实现阶段

当前代码阶段已完成 `Tauri + React + TypeScript + Vite` 基础骨架，目标如下：

1. 已完成设置页、Todo 主工作台、会话查看页与前端本地持久化原型。
2. 已接入 Tauri 桌面容器与最小 Rust 命令桥接。
3. 已接入 SQLite 的 `app_settings`、`audio_segments`、`conversation_sessions`、`todos`、`processing_jobs` 基础表结构。
4. 已实现设置读取/保存、Todo 状态切换、会话列表读取、手动刷新当前会话命令。
5. 已实现录音状态切换、模拟有效切片、模拟静默切片，以及基于静默切片的空闲触发会话骨架。
6. 已接入真实麦克风录音的第一步实现：Rust 侧改为“后台命令线程持有音频流 + 主线程通过命令控制开始/停止”，避免将音频流对象直接暴露在 Tauri 全局状态中。
7. 已补充双模型“测试连接”能力：设置页可直接对当前填写的 ASR / Todo 模型配置发起真实探测请求，便于排查地址、模型名与密钥问题。
8. ASR 配置已按火山语音识别能力拆分为 `submit_url + query_url + resource_id + model_name`，测试逻辑改为真实 `submit -> query` 两段式调用。
9. 当前真实录音会在停止时落地为本地 `wav` 文件，并写入 `audio_segments` 与 `processing_jobs`。
10. 已补齐“本地录音文件 -> ASR -> transcript_segments -> conversation_sessions -> Todo Extraction -> todos”的业务闭环。
11. 已增加基础稳定性保护：网络请求失败自动重试、429/5xx 退避重试、空会话/占位会话跳过提取、手动刷新冷却控制。
12. 下一阶段优先落地“本地嵌入 Todo 模型”，采用子进程推理，本阶段暂不内嵌本地 ASR。

## 4. 核心模块设计

### 4.1 设置与配置模块

职责：

1. 维护录音开关。
2. 维护 `chunk_seconds` 和 `idle_trigger_seconds`。
3. 维护双模型配置：
   `asr_base_url / asr_model_name / asr_api_key`
   `todo_base_url / todo_model_name / todo_api_key`
4. 负责配置校验、保存、加载。
5. 维护 `asr_provider_type`、`todo_provider_type` 以及本地 Todo 运行时状态。

关键约束：

1. 密钥不明文回显。
2. `chunk_seconds`、`idle_trigger_seconds` 必须为正整数。
3. 运行时任务从当前生效配置读取参数。
4. 本地 Todo 模型状态与云端配置需解耦，避免把子进程细节直接暴露给 UI。

### 4.2 音频采集模块

职责：

1. 持续采集环境音。
2. 每 `30 秒` 默认切片。
3. 为每个切片生成音频元数据记录。
4. 检测是否存在“有效录音”。

输入：

1. 麦克风输入流。
2. 设置项中的录音开关和切片参数。

输出：

1. `audio_segments` 记录。
2. 待转写任务。

### 4.3 语音转写模块

职责：

1. 将音频片段送入 ASR Provider。
2. 获取中文文本。
3. 写入 `transcript_segments`。
4. 失败时写入处理任务状态。

关键设计：

1. 每个音频片段独立转写。
2. 转写成功后不直接产出 Todo。
3. 统一返回结构，避免不同供应商格式污染业务层。

### 4.4 会话聚合模块

职责：

1. 监听新的文本片段写入。
2. 判断会话是否仍在进行中。
3. 当连续 `20 秒` 未识别到有效录音时，将最近文本片段合并为会话文稿。
4. 创建 `conversation_sessions` 记录。

关键设计：

1. 会话聚合不依赖单个音频切片边界。
2. 聚合逻辑只看“有效录音时间差”和文本片段时间顺序。
3. 后续支持手动强制刷新。

### 4.5 Todo 提取模块

职责：

1. 将会话文稿送入 Todo Extraction Provider。
2. 解析结构化返回结果。
3. 写入多条 `todos`。
4. 失败时保留原始会话文稿与任务状态。

关键设计：

1. 一次会话可以生成多条 Todo。
2. 必须保存 `conversation_session_id` 以便追溯来源。
3. 当 `todo_provider_type=embedded_local` 时，通过本地模型子进程完成结构化提取。
4. 本地模型输出必须通过 JSON schema 或等价校验后才能入库。
5. 是否允许云端兜底由 `allow_cloud_fallback` 控制，关闭后本地提取失败应直接暴露失败状态而不是自动切换云端。

### 4.5.1 本地模型运行时模块

职责：

1. 管理模型文件目录、版本与初始化状态。
2. 启动、停止并监控本地推理子进程。
3. 为 Todo 提取模块提供健康检查、超时控制和失败重试策略。

关键设计：

1. 模型资产随应用分发，但运行时文件释放到本地应用数据目录。
2. 子进程对外暴露稳定的本地调用接口，业务层不直接感知具体推理引擎。
3. 运行时异常要有明确状态：`not_ready`、`starting`、`ready`、`failed`。

### 4.6 Todo 管理模块

职责：

1. 提供 Todo 查询、详情、状态切换、编辑。
2. 展示来源会话和原始文稿。
3. 支持按状态筛选。

## 5. 核心任务流

### 5.1 主任务流

```text
应用启动
-> 读取 app_settings
-> record_enabled = true
-> 开始录音
-> 30 秒切片
-> 创建 audio_segments
-> 调用 ASR Provider
-> 写入 transcript_segments
-> 更新最后有效录音时间
-> 若连续 20 秒无有效录音
-> 创建 conversation_sessions
-> 调用 Todo Extraction Provider
-> 写入 todos
-> 更新 processing_jobs
```

### 5.2 异常任务流

```text
转写失败
-> transcript_segments.status = failed
-> processing_jobs.status = failed
-> 保留 audio_segments
-> 对网络异常与 429/5xx 做有限次重试
-> 达到上限后等待人工或后续自动重试

Todo 提取失败
-> conversation_sessions.extraction_status = failed
-> processing_jobs.status = failed
-> 保留 merged_text
-> 对网络异常与 429/5xx 做有限次重试
-> 支持手动或自动重试
-> 记录 extraction_provider_used、extraction_fallback_used 与 extraction_fallback_reason 便于排障

本地模型不可用 / 输出非法
-> local runtime status = failed
-> conversation_sessions.extraction_status = failed
-> processing_jobs.status = failed
-> 保留 merged_text 与模型响应摘要
-> 若 `allow_cloud_fallback=true`，则切换到云端 Todo Provider 重试
-> 若 `allow_cloud_fallback=false`，则保持失败并返回明确失败原因

空会话 / 占位会话
-> 不调用 Todo Provider
-> conversation_sessions.extraction_status = success
-> processing_jobs.status = success
-> 结果计为 0 条 Todo
```

## 6. 状态机设计

### 6.1 录音状态机

| 状态 | 说明 | 可迁移到 |
| --- | --- | --- |
| idle | 未录音 | recording |
| recording | 正在采集音频 | paused, stopped |
| paused | 暂停录音 | recording, stopped |
| stopped | 已停止 | recording |

### 6.2 会话状态机

| 状态 | 说明 | 可迁移到 |
| --- | --- | --- |
| collecting | 正在积累文本片段 | idle_waiting |
| idle_waiting | 等待是否继续有有效录音 | collecting, ready_for_extraction |
| ready_for_extraction | 满足提取条件 | extracting |
| extracting | 正在提取 Todo | extracted, failed |
| extracted | 提取完成 | - |
| failed | 提取失败 | ready_for_extraction |

### 6.4 本地模型运行时状态机

| 状态 | 说明 | 可迁移到 |
| --- | --- | --- |
| not_ready | 模型未准备好 | starting |
| starting | 正在启动或准备模型运行时 | ready, failed |
| ready | 本地模型可调用 | failed |
| failed | 本地模型不可用 | starting |

## 7. 本地嵌入模型设计

### 7.1 设计范围

1. 第一阶段仅内嵌 Todo 提取模型。
2. 采用“应用内置模型资产 + 子进程推理”方案。
3. 保留云端 Todo Provider 作为兜底。
4. 本阶段不实现本地 ASR。

### 7.2 运行时方案

```text
Tauri 主进程
-> ModelRuntimeManager
-> 启动本地 Todo 模型子进程
-> 健康检查通过
-> TodoExtractionService 调用本地 Provider
-> 返回结构化 JSON
-> 校验通过后写入 todos
```

设计要点：

1. 子进程必须独立于主进程，避免推理阻塞录音、数据库和 UI。
2. 模型文件建议首次启动后释放到应用数据目录，而非长期直接读取 App 包内文件。
3. 模型资产需随应用二进制分发，并在释放后进行版本与完整性校验。
4. 本地模型调用必须具备超时、重试、失败回退和状态上报。

### 7.3 演进策略

1. 第一阶段：内嵌 Todo 模型。
2. 第二阶段：根据验证结果评估本地 ASR。
3. 第三阶段：再考虑模型版本升级、热更新和多模型切换。

### 6.3 Todo 状态机

| 状态 | 说明 | 可迁移到 |
| --- | --- | --- |
| pending | 未完成 | completed |
| completed | 已完成 | pending |

## 7. 数据库设计映射

### 7.1 核心表

| 表名 | 作用 |
| --- | --- |
| `app_settings` | 保存应用配置与双模型配置 |
| `audio_segments` | 保存音频切片元数据 |
| `transcript_segments` | 保存转写结果 |
| `conversation_sessions` | 保存会话级文稿 |
| `todos` | 保存待办结果 |
| `processing_jobs` | 保存任务状态与失败信息 |

### 7.2 关键关系

1. `audio_segments.id -> transcript_segments.audio_segment_id`
2. `conversation_sessions.id -> transcript_segments.conversation_session_id`
3. `conversation_sessions.id -> todos.conversation_session_id`
4. `conversation_sessions.id -> processing_jobs.target_id`

## 8. Provider 抽象设计

### 8.1 ASR Provider 接口

输入：

```json
{
  "audio_path": "/tmp/audio_001.wav",
  "language": "zh-CN",
  "trace_id": "trace_001"
}
```

输出：

```json
{
  "success": true,
  "text": "下午给客户发送报价",
  "confidence_score": 0.95,
  "raw_response": {}
}
```

### 8.2 Todo Extraction Provider 接口

输入：

```json
{
  "merged_text": "下午给客户发送报价，然后确认税率口径。",
  "trace_id": "trace_001"
}
```

输出：

```json
{
  "success": true,
  "todos": [
    {
      "title": "给客户发送报价",
      "note": "确认税率口径"
    }
  ],
  "raw_response": {}
}
```

### 8.3 设计要求

1. Provider 层必须统一返回结构。
2. 原始响应建议保留在调试字段中，便于排障。
3. 业务层不得直接依赖供应商特定字段。

## 9. 接口与服务设计

### 9.1 内部服务建议

| 服务名 | 核心职责 |
| --- | --- |
| `SettingsService` | 配置读取、保存、校验 |
| `RecordingService` | 录音控制、切片调度 |
| `AudioSegmentService` | 音频片段记录持久化 |
| `TranscriptionService` | 触发转写并写入文本片段 |
| `ConversationService` | 聚合文本片段并生成会话 |
| `TodoExtractionService` | 从会话提取 Todo |
| `TodoService` | Todo 查询、更新、状态切换 |
| `ProcessingJobService` | 任务状态更新、重试管理 |

## 10. 并发与调度建议

1. 录音采集和转写应异步解耦。
2. 转写任务可按片段串行或小并发执行，避免瞬时请求过多。
3. 会话聚合应使用单独调度器，避免在 UI 线程完成。
4. Todo 提取应按会话粒度执行，避免重复提取。

## 11. 安全设计

1. 模型密钥优先存 Keychain，不建议明文入 SQLite。
2. 日志中不得输出完整音频路径、完整文稿和完整密钥。
3. 所有配置读取接口默认返回脱敏字段。
4. 当模型配置缺失时，直接阻断相关任务执行。

## 12. 可观测性设计

建议统一使用 `trace_id` 贯穿以下阶段：

1. 音频切片创建
2. 转写请求
3. 会话聚合
4. Todo 提取
5. Todo 入库

建议日志字段：

1. `trace_id`
2. `job_type`
3. `target_id`
4. `status`
5. `error_message`

## 13. 一期实现建议

### 13.1 建议优先级

1. 先做设置页与 `app_settings` 落库。
2. 再打通音频切片和 `audio_segments`。
3. 接着接入 ASR Provider，写入 `transcript_segments`。
4. 再实现会话聚合规则和 `conversation_sessions`。
5. 最后接入 Todo Extraction Provider 和 `todos`。

### 13.2 一期可简化点

1. `processing_jobs` 可先做轻量实现。
2. 手动刷新会话可先只做接口，不做复杂 UI。
3. 置信度与能量分数可先做可空字段。

## 14. 结论

一期技术设计的关键不在“实时字幕”，而在于把“分段转写”和“会话级 Todo 提取”拆开。只要这两个环节拆得清楚，后续接本地模型、做声纹识别、做失败重试都会顺很多。
