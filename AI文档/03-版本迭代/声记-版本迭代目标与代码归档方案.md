# 声记版本迭代目标与代码归档方案

文档状态：v1.0
生成日期：2026-06-12
适用项目：`shengji_mac`
当前基线版本：`0.3.0`
依据文档：`AI文档/02-技术方案/声记-版本迭代与项目架构方案.md`

## 1. 结论

后续版本按 `v0.4 -> v1.1` 推进。主线不是继续堆叠旧的“录音转 Todo”原型，而是把项目整理成“本地转写 + 说话人分离 + MiniMax M3 语义理解 + 统一 AI 产物”的桌面工作台。

代码归档采用两个原则：

1. **Git tag 是代码快照来源**：每个正式版本通过 `vX.Y.Z` tag 固化代码，版本显示名使用 `vX.Y.Z (YYYY.M.D)`，不在仓库里复制一份源码快照，避免重复代码失控。
2. **工作区只保留当前可维护代码**：旧实现先进入 `legacy` 边界，稳定两个版本后删除；旧文档、旧方案、发布说明进入 `AI文档` 的归档目录。

## 2. In Scope / Out of Scope

### In Scope

1. 重新梳理 `v0.4` 到 `v1.1` 的版本目标、交付内容和验收标准。
2. 规定每个版本结束时，代码、文档、分支、tag、发布说明如何整理归档。
3. 明确工作区源码目录演进规则，防止 `src/App.tsx` 和 `src-tauri/src/lib.rs` 继续变成单体文件。
4. 明确哪些内容禁止提交到 Git，例如本地录音、数据库、模型权重、运行缓存。

### Out of Scope

1. 不在本文档中重写完整 PRD。
2. 不在本文档中设计完整数据库 DDL。
3. 不在本文档中替代 `Git规范.md` 和 `开发规范.md`。
4. 不把每个版本的源码复制到 `AI文档` 或其他目录作为备份。
5. 暂不纳入播客化、播客脚本生成和 TTS 输出，后续单独评估再迭代。

## 4. 工作区归档总规则

### 4.1 源码归档

源码不按版本复制目录。每个版本结束时用以下方式归档：

1. 按 `feature-* / hotfix-* -> dev -> test -> uat -> release -> main` 合版本。
2. 在 `main` 发布合并提交上打 `vX.Y.Z` tag。
3. 保留发布说明和归档清单。
4. 删除已合并临时分支。

推荐命令：

```bash
git checkout dev
git merge --no-ff feature-vX.Y.Z-主题
git checkout test
git merge --no-ff dev
git checkout uat
git merge --no-ff test
git checkout release
git merge --no-ff uat
git checkout main
git merge --no-ff release
git tag -a vX.Y.Z -m "Release vX.Y.Z (YYYY.M.D)"
```

如果需要查看历史版本代码，使用：

```bash
git checkout vX.Y.Z
```

### 4.2 文档归档

每个版本结束后补齐：

```text
AI文档/
├── 发布说明_vX.Y.Z.md
└── 版本归档/
    └── vX.Y.Z/
        ├── 归档清单.md
        ├── 验收记录.md
        └── 已知问题.md
```

如果某个旧方案、旧设计或旧规范已被替代，移动到：

```text
AI文档/废纸篓/YYYY-MM-DD-vX.Y.Z-旧方案归档/
```

### 4.3 代码目录整理

每个版本结束时，源码必须只保留三类内容：

1. **当前主线代码**：仍被默认路径使用。
2. **实验代码**：必须有 feature flag 或明确入口，不进入默认路径。
3. **legacy 代码**：只为回滚保留，必须标注计划删除版本。

推荐标注格式：

```rust
// LEGACY(v0.4): 保留旧本地 LLM Todo 提取路径用于回滚，计划 v0.6 后删除。
```

### 4.4 禁止提交内容

以下内容不进入 Git：

1. 本地录音文件。
2. SQLite 数据库和数据库备份。
3. `.gguf`、WhisperKit 模型权重、SpeakerKit 模型权重。
4. `argmax-cli` 二进制和本地运行缓存。
5. API Key、完整用户转写文本、完整音频路径。
6. `dist/`、`build/`、`.tauri/`、`.vite/`、`.DS_Store` 等生成物。

模型、sidecar、安装包如需发布，放 GitHub Release 附件，不放源码仓库。

### 4.5 每版最小验证

每个版本结束前至少执行：

```bash
npm run build
cargo check --manifest-path src-tauri/Cargo.toml
```

涉及工具脚本时追加：

```bash
npm run test:tools
```

如果某项无法执行，必须写入 `AI文档/版本归档/vX.Y.Z/已知问题.md`。

## 5. v0.4 架构止血与本地 ASR 路线切换

### 目标

停止继续向旧单体架构堆功能，为“本地 ASR + 本地说话人分离 + MiniMax M3 语义理解”建立清晰边界。

### 交付内容

1. 拆分后端目录：

```text
src-tauri/src/
├── commands/
├── app/
├── domain/
├── infra/
├── providers/
└── jobs/
```

2. 新增 provider 抽象：
   - `AsrProvider`
   - `SpeakerProvider`
   - `SemanticProvider`
   - `EmbeddingProvider`，仅预留接口
   - `ExportProvider`，仅预留接口
3. 新增 `semantic_artifacts` 与 `model_invocations`。
4. MiniMax M3 只作为语义 provider，不作为 ASR 主线。
5. 旧本地 Qwen / llama.cpp Todo 路径标记为 `legacy`。

### 代码整理与归档

| 区域 | 整理方式 |
| --- | --- |
| `src-tauri/src/lib.rs` | 只保留 Tauri 初始化、state 注入、command 注册，不继续承载业务逻辑 |
| `src-tauri/src/commands/` | 放 Tauri command，负责参数校验和 DTO 返回 |
| `src-tauri/src/app/` | 放业务服务，例如 transcription、speaker、semantic、todo |
| `src-tauri/src/domain/` | 放实体、状态枚举、错误类型 |
| `src-tauri/src/infra/` | 放 SQLite、HTTP、Keychain、进程管理、音频文件 |
| `src-tauri/src/providers/` | 放 ASR、Speaker、Semantic provider 实现 |
| `src-tauri/src/jobs/` | 放后台任务、队列、重试状态机 |
| 旧本地 LLM 代码 | 移入 `providers/semantic/legacy_local_llm.rs` 或同等 legacy 模块 |
| 旧方案文档 | 已替代内容移动到 `AI文档/废纸篓/2026-06-12-v0.4-旧方案归档/` |

### 验收标准

1. `npm run build` 通过。
2. `cargo check --manifest-path src-tauri/Cargo.toml` 通过。
3. `AsrProvider`、`SpeakerProvider`、`SemanticProvider` 可以独立配置。
4. Todo 提取不再直接依赖旧本地 LLM 运行时。
5. 设置页能说明“本地 ASR / 本地说话人分离 / 云端语义模型”的隐私边界。

2. 支持录音切片任务化。
3. 接入 `argmax-cli` local server 或 CLI。
4. 新增本地模型管理：下载进度、缓存目录、离线状态、设备推荐。
5. 新增转写任务状态机。
6. 新增 `transcript_segments`、`speakers`、`speaker_segments`。
7. 新增转写评估页：音频播放、时间跳转、说话人标签、错误片段标注。

### 代码整理与归档

| 区域 | 整理方式 |
| --- | --- |
| `src-tauri/src/providers/asr/local_whisperkit.rs` | 放 WhisperKit / Argmax 转写 provider |
| `src-tauri/src/providers/asr/local_argmax_server.rs` | 放 local server 健康检查和 API 适配 |
| `src-tauri/src/providers/speaker/local_speakerkit.rs` | 放说话人分离 provider |
| `src-tauri/src/infra/process/` | 放 sidecar 进程启动、停止、端口检查 |

| `src/views/TranscriptView.tsx` | 放转写评估页 |
| `src/components/AudioPlayer.tsx` | 放音频播放和时间跳转 |
| `src/components/TranscriptTimeline.tsx` | 放时间轴转写列表 |
| 本地模型文件 | 不进 Git，只记录 manifest、缓存目录说明和下载策略 |

### 验收标准

1. 一段音频能生成时间轴转写稿。
2. 点击转写片段能跳到对应音频时间。
3. 说话人 label 可重命名并持久化。
4. 失败任务可以重试。
5. 断网状态下可完成本地转写。
6. 说话人分离错误可由用户手动修正。

2. 摘要能追溯到转写片段。

## 9. v0.8 思维脑图

### 代码整理与归档

| --- | --- |

## 10. v0.9 Moment 与深度研究

4. 不出现播客脚本、TTS provider 或音频生成入口。

## 13. 每个版本的收尾清单

每个版本完成后按以下顺序执行。

### 13.1 代码整理

1. 删除未使用的临时代码和调试入口。
2. 将仍需回滚的旧路径标记为 `legacy`，写明计划删除版本。
3. 确认业务逻辑没有重新堆回 `src/App.tsx` 或 `src-tauri/src/lib.rs`。
4. 确认 provider 请求不在前端直接拼装。
5. 确认模型权重、录音、数据库、缓存没有进入 Git。

### 13.2 文档整理

1. 新增或更新 `AI文档/04-发布记录/发布说明_vX.Y.Z.md`。
2. 新增 `AI文档/版本归档/vX.Y.Z/归档清单.md`。
3. 新增 `AI文档/版本归档/vX.Y.Z/验收记录.md`。
4. 如有未解决问题，新增 `AI文档/版本归档/vX.Y.Z/已知问题.md`。
5. 被替代的旧方案移动到 `AI文档/废纸篓/YYYY-MM-DD-vX.Y.Z-旧方案归档/`。

### 13.3 版本一致性

发版前同步检查：

| 文件 / 对象 | 要求 |
| --- | --- |
| `package.json` | `version` 等于目标版本 |
| `src-tauri/Cargo.toml` | `version` 等于目标版本 |
| `src-tauri/tauri.conf.json` | `version` 等于目标版本 |
| `AI文档/04-发布记录/发布说明_vX.Y.Z.md` | 文件名、正文版本和发布日期一致 |
| Git tag | `vX.Y.Z` |
| Release 标题 | `vX.Y.Z (YYYY.M.D)` |

### 13.4 验证记录

把以下内容写入 `验收记录.md`：

1. 执行的验证命令。
2. 命令结果。
3. 手动验证路径。
4. 未验证项。
5. 已知风险和回滚方式。

### 13.5 分支、合版本与 tag

1. 开发分支命名：`feature-vX.Y.Z-主题`。
2. 紧急修复分支命名：`hotfix-vX.Y.Z-主题`。
3. 每个版本开发完成后，必须按 `feature-* / hotfix-* -> dev -> test -> uat -> release -> main` 晋升，这一步称为合版本。
4. 合并策略：`git merge --no-ff`。
5. 合版本后在 `main` 上重新执行最小验证。
6. 正式 tag 必须打在 `main` 的发布合并提交上，tag 格式：`vX.Y.Z`。
7. Release 和发布说明标题必须使用显示名：`vX.Y.Z (YYYY.M.D)`，例如 `v0.16.0 (2026.6.5)`。
8. 已合并分支默认删除。
9. 已发布 tag 不改指向；问题通过补丁版本修复，例如 `v1.0.1`。

推荐合版本命令：

```bash
git checkout dev
git pull origin dev
git merge --no-ff feature-vX.Y.Z-主题
npm run build
cargo check --manifest-path src-tauri/Cargo.toml

git checkout test
git pull origin test
git merge --no-ff dev

git checkout uat
git pull origin uat
git merge --no-ff test

git checkout release
git pull origin release
git merge --no-ff uat

git checkout main
git pull origin main
git merge --no-ff release
git tag -a vX.Y.Z -m "Release vX.Y.Z (YYYY.M.D)"

git push origin dev test uat release main
git push origin vX.Y.Z
```

## 15. 回滚规则

1. 未发布版本：优先追加修复提交，不做破坏性回滚。
2. 已发布版本：不改旧 tag，不覆盖 Release，通过补丁版本修复。
3. legacy 代码只保留到约定版本，不能无限期存在。
4. 数据库 migration 一旦进入发布版本，禁止修改历史 migration，只能新增修复 migration。
5. 如果 provider 路线失败，回滚 provider 实现，不回滚 domain 和 artifact 主结构。
