# 开发规范

## 1. 文档目标

本文档用于约束智能 Todo 项目在设计、开发、联调、测试与文档维护过程中的统一规范，确保一期功能可以稳定落地，并为后续本地模型、声纹识别等能力演进预留空间。

## 2. 总体原则

1. 优先保证一期闭环可用，再考虑能力扩展。
2. 优先模块解耦，避免录音、转写、Todo 提取、数据存储强耦合。
3. 所有解释性文案、界面文案、日志说明优先使用中文。
4. 所有文档与代码文件统一使用 UTF-8 无 BOM 编码。
5. 涉及用户隐私的音频、文本、密钥配置，默认按敏感数据处理。

## 3. 架构规范

### 3.1 模块拆分

项目至少拆分为以下模块：

1. 设置与配置模块。
2. 音频采集模块。
3. 语音转写模块。
4. 会话聚合模块。
5. Todo 提取模块。
6. Todo 管理模块。
7. 数据存储模块。
8. AI Provider 抽象层。

### 3.2 分层要求

1. UI 层只负责展示、交互和状态分发，不直接写 AI 调用逻辑。
2. 应用服务层负责业务编排，例如“录音切片后触发转写，转写结果进入会话缓冲区，满足空闲条件后再触发 Todo 提取”。
3. 基础设施层负责本地数据库、音频设备、网络请求、密钥存取等能力。
4. AI Provider 必须通过统一接口接入，禁止在业务层硬编码某一家模型服务。
5. 语音转写模型与 Todo 提取模型必须独立配置，禁止共用单一硬编码配置。

## 4. 数据与配置规范

### 4.1 Todo 数据规范

每条 Todo 至少包含以下字段：

1. `id`
2. `title`
3. `note`
4. `created_at`
5. `status`

建议预留以下扩展字段：

1. `source_text`
2. `source_audio_id`
3. `speaker_id`
4. `llm_provider`
5. `trace_id`

建议新增以下中间对象：

1. `transcript_segment`
2. `conversation_session`

### 4.2 配置项规范

一期至少支持以下配置项：

1. `record_enabled`
2. `chunk_seconds`，默认值 `30`
3. `idle_trigger_seconds`，默认值 `20`
4. `language`
5. `provider_mode`
6. `asr_api_key`
7. `asr_base_url`
8. `asr_model_name`
9. `todo_api_key`
10. `todo_base_url`
11. `todo_model_name`

### 4.3 敏感信息规范

1. `asr_api_key` 和 `todo_api_key` 不允许明文输出到普通日志。
2. 敏感配置优先使用系统安全存储方案，避免直接明文落本地文件。
3. 调试时如需打印请求信息，必须脱敏处理。

## 5. 编码规范

### 5.1 通用要求

1. 优先使用清晰命名，避免缩写和模糊字段名。
2. 业务逻辑函数应单一职责，避免一个函数同时处理录音、转写、提取、入库。
3. 关键链路必须有错误处理和状态回传。
4. 复杂逻辑可以加简短注释，但禁止堆砌无意义注释。

### 5.2 接口与抽象

1. 语音转写能力必须抽象成独立接口。
2. Todo 提取能力必须抽象成独立接口。
3. 会话聚合逻辑必须抽象成独立服务，不允许散落在 UI 层和 Provider 层。
4. 两个云端模型接入都必须支持替换 `base_url`、`api_key`、`model_name`。
5. Provider 层应统一输入输出结构，避免上层适配多套返回格式。

### 5.3 状态管理

1. 录音状态、转写状态、提取状态、入库状态需可追踪。
2. 异步流程必须有明确状态枚举，禁止只靠布尔值拼装复杂状态。
3. UI 不直接依赖底层 SDK 返回值，统一通过 ViewModel 或状态对象暴露。
4. 会话状态至少区分：`collecting`、`idle_waiting`、`ready_for_extraction`、`extracted`、`failed`。

### 5.4 音频与会话策略

1. 一期默认音频切片时长为 `30` 秒。
2. 一期默认无有效录音触发间歇时间为 `20` 秒。
3. 每个音频片段应优先独立转写为文本片段，不直接从单片段生成 Todo。
4. 当连续 `20` 秒未识别到有效录音时，系统应将最近一段时间的文本片段合并为会话文稿，再触发 Todo 提取。
5. 切片时长和触发间歇时间必须支持在设置中修改，并在运行时读取最新配置。

## 6. 日志与可观测性规范

1. 每一段音频处理链路建议生成 `trace_id`。
2. 至少记录以下节点：开始录音、切片完成、转写成功/失败、会话聚合触发、Todo 提取成功/失败、入库成功/失败。
3. 错误日志必须包含错误阶段、错误类型、重试建议。
4. 日志中不得输出完整音频内容、完整密钥和其他敏感信息。

## 7. 安全与隐私规范

1. 录音必须由用户显式开启，并在界面中持续展示录音状态。
2. 未完成模型配置时，不允许自动发起云端 AI 调用。
3. 后续若支持音频上传，需明确上传范围、保留周期与删除策略。
4. 声纹识别上线前，不得提前采集和保存额外生物特征数据。

## 8. 测试与验证规范

1. 每次代码变更后必须执行最小可行验证。
2. 核心验证至少覆盖：录音开关、参数配置保存、双模型配置保存、转写调用、会话聚合触发、Todo 提取、Todo 状态切换。
3. 关键解析逻辑应补单元测试，尤其是模型返回结构解析和异常分支。
4. 提交前必须检查文档可读性、中文乱码、配置字段命名一致性。

## 9. 文档同步规范

出现以下变更时，必须同步更新 `AI文档` 下对应文档：

1. 产品需求变化，更新 PRD。
2. 技术架构变化，补充设计文档。
3. 数据结构变化，补充表结构或对象定义文档。
4. 接口契约变化，补充接口说明文档。

## 10. 一期开发边界

### In Scope

1. 录音开关与模型配置。
2. 切片时长与无有效录音触发间歇时间配置。
3. 环境音采集与分段。
4. 双模型云端转写与 Todo 提取链路。
5. 会话文稿聚合。
6. Todo 自动生成与状态切换。
7. 本地持久化存储。

### Out of Scope

1. 本地大模型正式接入。
2. 声纹识别。
3. 多端同步。
4. 自动提醒和日历集成。

## 11. 结论

一期开发规范的核心是四点：边界清晰、会话聚合优先、敏感数据安全、AI 能力可替换。所有实现都应围绕“先跑通分段转写与空闲触发 Todo 闭环，再做增强”的原则推进。
