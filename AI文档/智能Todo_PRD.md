## 2. 需求背景

### 2.1 问题背景

用户在电脑前长时间工作，很多待办事项来自临时对话、自言自语、会议讨论或环境中的即时口头信息。这类信息具备以下特点：

1. 产生频繁，且往往发生在用户正在处理其他事务时。
2. 用户不一定会立刻手动记录，导致待办遗漏。
3. 传统 Todo 工具依赖手动输入，录入成本高，难以覆盖碎片化语音场景。
4. 用户希望电脑作为长期陪伴式工作终端，持续监听并自动沉淀可执行事项。

### 2.2 机会判断

如果软件在开启后可持续录制环境音，并将语音自动转写为中文，再进一步提炼为结构化 Todo，则可显著降低记录成本，提高任务捕获率，并为后续个人智能助理能力打下基础。

## 3. 需求目标

### 3.1 产品目标

构建一款运行在电脑端的智能 Todo 软件，在用户授权和配置开启的前提下，实现“环境音采集 -> 语音转文字 -> Todo 提取 -> Todo 管理”的闭环。

### 3.2 一期目标

一期聚焦最小可用闭环，目标如下：

1. 支持配置是否开启环境音录制。
2. 支持在设置中配置音频切片时长与无有效录音触发间歇时间，默认分别为 30 秒和 20 秒。
3. 支持在设置中配置两个云端大模型，分别用于语音转文字和 Todo 提取，包括密钥、调用地址、模型类型。
4. 支持将录制音频自动转换为中文文本。
5. 支持在未持续识别到有效录音时，将最近一段时间的文稿合并后提取 Todo。
6. 每个 Todo 至少包含：标题、备注、创建时间、完成状态。
7. 支持 Todo 在“未完成 / 已完成”之间切换。
8. 支持后续扩展本地模型、声纹识别、指定用户过滤等能力的技术预留。

### 3.3 非目标

以下内容不纳入一期范围：

1. 本地大模型推理能力正式上线。
2. 声纹识别与指定用户过滤。
3. 多设备同步。
4. 自动日历排期、提醒推送、优先级推荐。
5. 多语言识别，当前仅聚焦中文。

## 4. 核心假设

1. 用户默认接受软件在开启且显式配置后录制环境音。
2. 一期优先在联网环境下运行，语音识别与 Todo 提取可依赖云端服务。
3. 环境音场景以普通办公场景为主，不追求极端噪声环境下的高准确率。
4. Todo 提取以“实用可用”优先，允许少量误识别，通过人工编辑和完成状态切换兜底。

## 5. 用户画像与典型场景

### 5.1 目标用户

1. 长时间在电脑前工作的知识工作者。
2. 经常边沟通边处理事务、容易遗漏口头待办的人群。
3. 愿意尝试 AI 辅助记录与整理的早期用户。

### 5.2 典型场景

1. 用户在开会时口头提到“下午把合同发给小李，顺便确认回款时间”，系统自动生成两个 Todo。
2. 用户工作时自言自语“明天要把录音模块的异常重试补上”，系统自动提取为待办。
3. 用户和同事对话中提到多个后续动作，系统自动拆分为多个 Todo，用户稍后统一确认和管理。

## 6. 方案说明

### 6.1 总体方案

一期采用“本地录音 + 云端语音转写/大模型提取 + 本地 Todo 管理”的方案，优先验证用户价值与识别链路闭环。

核心流程如下：

1. 用户开启应用，并在设置中打开“环境音录制”。
2. 用户在设置中填写切片时长、无有效录音触发间歇时间，以及两个云端大模型配置。
3. 应用持续采集环境音，并按 30 秒默认切片规则生成音频片段。
4. 每个音频片段送入语音转写模型，输出中文文本片段。
5. 当连续 20 秒未识别到有效录音时，系统将最近一段时间的文本片段合并为会话文稿。
6. 会话文稿送入 Todo 提取模型，生成一个或多个结构化 Todo。
7. Todo 写入本地列表，默认状态为“未完成”。
8. 用户可查看、编辑、标记完成或恢复未完成。

### 6.2 关键方案对比

| 方案 | 描述 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| 方案 A：纯云端链路 | 音频上传云端，完成转写与 Todo 提取 | 开发快、效果较稳、前期成本低 | 依赖网络、隐私敏感、云成本存在 | 一期推荐 |
| 方案 B：本地转写 + 云端提取 | 语音在本地转文字，文本上传云端提取 Todo | 降低音频上云敏感度 | 本地 ASR 集成复杂、设备性能要求更高 | 二期可评估 |
| 方案 C：纯本地链路 | 本地录音、本地转写、本地大模型提取 | 隐私更强、可离线 | 开发与调优成本高、模型部署重 | 长期方向，不做一期 |

### 6.3 一期推荐落地

一期建议采用方案 A，并做以下可扩展设计：

1. 录音模块、转写模块、Todo 提取模块解耦。
2. 统一抽象 AI Provider（AI 服务提供方）接口，后续可切换云端或本地模型。
3. Todo 数据模型预留 `speaker_id`、`source_text`、`source_audio_id` 等扩展字段。

## 7. 功能流程说明

### 7.1 主流程

```text
应用启动
-> 读取录音配置
-> 用户已开启录音
-> 开始环境音采集
-> 音频分段
-> 语音转中文
-> 大模型提取 Todo
-> 写入 Todo 列表
-> 用户查看 / 编辑 / 完成切换
```

### 7.2 异常流程

1. 用户未开启录音：系统不采集音频，仅展示关闭状态。
2. 用户未完成模型配置：系统不发起对应 AI 处理，并提示补全配置。
3. 音频为空或识别失败：记录失败任务，不生成文本片段。
4. 长时间未达到有效录音阈值：仅缓存录音状态，不触发 Todo 提取。
5. 文本无法提取有效 Todo：保留原始会话文稿，标记“未提取”供后续人工处理。
6. 云端服务异常：提示处理失败并支持后续重试。
7. 手动刷新过于频繁：触发冷却提示，不重复创建无效会话。
8. 占位会话或空文稿：直接跳过 Todo 提取，避免制造无意义失败记录。

## 8. 系统规划

### 8.1 系统模块划分

| 模块 | 职责 |
| --- | --- |
| 设置与配置模块 | 控制是否开启环境音录制、维护切片与间歇参数、维护两个云端模型配置、展示服务状态、处理基础权限管理 |
| 音频采集模块 | 持续采集环境音、按配置切片、缓存音频片段、判断有效录音 |
| 语音转写模块 | 调用第一模型将音频片段转换为中文文本片段 |
| 会话聚合模块 | 按时间窗口合并最近文本片段，并基于无有效录音间歇时间触发会话结束 |
| Todo 提取模块 | 调用第二模型从会话文稿中提取多个结构化 Todo |
| Todo 管理模块 | Todo 列表展示、详情编辑、完成状态切换 |
| 数据存储模块 | 保存设置配置、音频片段元数据、文本片段、会话文稿、Todo 数据与处理状态 |
| AI Provider 抽象层 | 封装云端 / 本地模型接入差异 |

### 8.3 数据表设计

为保证一期可落地、可追踪、可回溯，建议本地存储采用 SQLite，并按如下核心表设计。

#### 8.3.1 表关系概览

```text
app_settings
audio_segments 1 -> N transcript_segments
conversation_sessions 1 -> N transcript_segments
conversation_sessions 1 -> N todos
conversation_sessions 1 -> N processing_jobs
```

说明：

1. `audio_segments` 表示原始音频切片元数据。
2. `transcript_segments` 表示每个音频切片转写后的文本片段。
3. `conversation_sessions` 表示按“20 秒无有效录音”聚合出的会话文稿。
4. `todos` 表示最终从会话文稿中提取出来的待办。
5. `processing_jobs` 用于记录转写、聚合、提取等任务状态，便于排障和重试。
6. `app_settings` 保存非敏感设置；密钥建议优先存系统 Keychain，仅在表中保存引用标识。

用途：保存应用级设置项与模型接入配置。

| 字段 | 类型 | 主键/索引 | 说明 |
| --- | --- | --- | --- |
| id | TEXT | PK | 固定单行主键，如 `default` |
| record_enabled | INTEGER |  | 是否开启录音，`0/1` |
| language | TEXT |  | 当前默认 `zh-CN` |
| chunk_seconds | INTEGER |  | 切片时长，默认 `30` |
| idle_trigger_seconds | INTEGER |  | 无有效录音触发时长，默认 `20` |
| provider_mode | TEXT |  | 当前一期固定 `cloud` |
| asr_base_url | TEXT |  | 语音转写模型调用地址 |
| asr_model_name | TEXT |  | 语音转写模型名称 |
| asr_api_key_ref | TEXT |  | 转写模型密钥在系统安全存储中的引用标识 |
| todo_base_url | TEXT |  | Todo 提取模型调用地址 |
| todo_model_name | TEXT |  | Todo 提取模型名称 |

### 8.4 技术演进规划

#### 第一阶段：闭环验证

1. 打通录音、转写、Todo 提取、Todo 管理。
2. 打通两个云端模型配置、鉴权与调用链路。
3. 打通“文本片段 -> 会话文稿 -> Todo”聚合策略。
4. 验证待办提取准确率与用户接受度。

#### 第二阶段：能力增强

1. 支持本地 ASR 或本地大模型提取。
2. 增加失败重试、历史记录与人工确认机制。

#### 第三阶段：身份过滤

1. 引入声纹识别。
2. 仅识别指定用户声音并生成 Todo。

## 10. 一期交互要求

### 10.1 最小界面范围

一期建议包含以下页面或区域：

1. 设置区：录音开关、服务状态、模型模式展示。
2. 参数配置区：支持填写切片时长和无有效录音触发间歇时间，并展示默认值与当前值。
3. 语音转写模型配置区：支持填写密钥、调用地址、模型类型，并展示当前配置状态。
4. Todo 提取模型配置区：支持填写密钥、调用地址、模型类型，并展示当前配置状态。
5. Todo 列表区：展示未完成 / 已完成事项。
6. Todo 详情区：展示标题、备注、创建时间。
7. 状态反馈区：展示“录音中 / 转写中 / 等待会话结束 / 提取成功 / 提取失败 / 配置缺失”等系统状态。

### 10.2 状态要求

1. 录音开启时需有明显状态标识，避免用户误解是否正在录音。
2. 切片时长与无有效录音触发间歇时间需支持数值校验，避免非法配置进入运行态。
3. 两个模型配置项需支持分别保存前校验，避免空值或非法地址直接进入运行态。
4. 新 Todo 生成后需即时出现在列表中。
5. 完成状态切换需即时反馈。

## 11. 验收标准

### 11.1 业务验收

1. 用户开启录音后，系统可持续处理环境音并生成 Todo。
2. 用户可在设置中成功保存切片时长与无有效录音触发间歇时间。
3. 用户可在设置中成功保存两个模型的密钥、调用地址、模型类型。
4. 对一段包含多个行动项的中文语音，系统在连续 20 秒无有效录音后可提取至少 2 条结构化 Todo。
5. 每条 Todo 均包含标题、备注、创建时间、完成状态。
6. 用户可手动切换 Todo 完成与未完成。

### 11.2 技术验收

1. 模块边界清晰，录音、转写、会话聚合、提取、存储相互解耦。
2. AI Provider 接口具备后续扩展到本地模型的能力。
3. 数据结构预留声纹识别字段，不阻塞二期扩展。

## 12. 风险与应对

| 风险 | 说明 | 应对方案 |
| --- | --- | --- |
| 隐私风险 | 环境音录制对用户较敏感 | 强制显式开关、清晰状态提示、后续补充隐私说明 |
| 识别准确率不足 | 噪声环境或口语表达导致提取不准 | 一期允许人工校正，后续优化 Prompt 与模型策略 |
| 云端依赖 | 网络或 API 异常影响可用性 | 增加失败重试与任务状态记录 |
| 成本风险 | 音频与模型调用成本可能偏高 | 做音频切片与过滤策略，控制上传频率 |

## 13. 结论

智能 Todo 一期应优先验证“持续环境音采集 -> 分段转写 -> 空闲触发会话聚合 -> 多 Todo 提取 -> Todo 管理”的核心闭环，采用双模型云端方案快速落地，并在架构上为本地模型与声纹识别预留清晰扩展点。该方案既能降低切片截断带来的理解偏差，也能为后续演进提供稳定基础。
