# 声记版本迭代与项目架构方案

文档状态：v1.0
生成日期：2026-06-12
适用项目：`shengji_mac`
当前代码版本：`0.3.0`

## 1. 结论

声记后续不建议继续沿用“本地小模型 + 单体 Tauri command + 简单 Todo 表”的路线。当前项目已经跑通一期闭环，但架构仍是工程验证原型，继续叠功能会让转写、摘要、Todo、脑图、深度研究全部挤在同一条脆弱链路里。

新的产品方向应调整为：

```text
录音 / 导入音频
-> 语音转写与说话人分离
-> 时间轴转写稿
-> MiniMax M3 转写修正与缺失补全
-> 修正文稿与本地修正记忆
-> MiniMax M3 类型识别与结构化处理
-> 摘要 / 纪要 / Todo / 思维脑图 / Moment / 深度研究 / 导出
```

新的技术方向应调整为：

1. 保留 Tauri 2 + Rust + React + SQLite 的桌面架构。
2. 暂停本地 LLM 主线，不再把 Qwen + llama.cpp 作为核心路线。
3. MiniMax M3 作为语义理解模型，用于转写修正、录音类型识别、摘要、纪要、Todo、脑图、深度研究、长上下文整合。
4. 语音转写必须独立为 `AsrProvider`，不要直接绑定 MiniMax M3。
5. ASR 输出不能直接进入摘要和 Todo 生成，必须先经过可追溯的转写修正流程。
6. 所有 AI 产物统一进入 `semantic_artifacts`，避免每新增一个功能就新增一套孤立流程。

## 2. 当前项目现状

### 2.1 已具备能力

当前项目已经具备桌面端基础闭环：

1. macOS 桌面端启动与本地数据持久化。
2. 麦克风录音并保存本地 WAV 文件。
3. 本地 SQLite 存储音频、转写、会话、Todo。
4. ASR 配置与模型配置界面。
5. 火山 ASR 接入路径。
6. Todo 提取任务与基础重试。
7. Todo 列表展示、完成状态切换。
8. 基础运行状态和失败原因记录。

### 2.2 主要不足

| 问题 | 现状 | 影响 |
| --- | --- | --- |
| 架构过于集中 | Rust 主要逻辑集中在 `src-tauri/src/lib.rs`，前端主要逻辑集中在 `src/App.tsx` | 新功能会继续堆叠，测试和维护成本快速上升 |
| 模型职责混乱 | ASR、Todo 提取、本地模型运行时混在同一业务链路里 | 难以切换 MiniMax M3，也难以保留多 provider |
| 产品层级偏低 | 当前核心是 Todo，不是完整语音知识工作台 | 后续摘要、脑图、研究、分享都缺少统一承载 |
| 数据模型太薄 | `todos` 和转写相关表不足以表达说话人、来源片段、置信度、AI 产物 | Todo 无法精确追溯，脑图和深度研究也缺存储基础 |
| 任务系统较弱 | 当前更像同步扫描 pending jobs | 缺少后台 runner、任务状态机、重试策略和可观测性 |
| 本地 LLM 路线收益低 | 内嵌 Qwen + llama.cpp 增加包体、兼容性和运行时问题 | 语义理解转向 MiniMax M3；本地能力保留给 ASR 和说话人分离 |

## 3. 产品定位

### 3.1 新定位

声记应定位为 macOS 上的“语音知识与行动工作台”，而不是单纯的录音转 Todo 工具。

用户核心流程：

1. 开始录音或导入音频。
2. 获得可播放、可跳转、可编辑的转写稿。
3. 识别说话人并允许重命名。
4. 通过 MiniMax M3 修正错识别、疑似漏字漏词和结构混乱片段。
5. 自动识别会议或录音类型，并生成摘要、纪要、待办、决策、问题、关键片段。
6. 生成思维脑图，辅助复盘和知识整理。
7. 将 Todo 进入行动中心，将知识产物导出或分享。

### 3.2 参照 TicNote 的功能推进

从 TicNote 分享页公开前端形态观察，目标体验包含：

| TicNote 模块 | 声记对应能力 | 建议版本 |
| --- | --- | --- |
| 音频播放 | 音频预览、时间跳转 | v0.5 |
| Transcript | 分段转写、时间轴 | v0.5 |
| Speaker | 说话人分离、说话人重命名 | v0.5 |
| Summary | 摘要、主题、结论 | v0.6 |
| Todo | 待办提取、确认、追溯 | v0.7 |
| MindMap | 思维脑图生成与编辑 | v0.8 |
| Moment | 关键片段、精彩时刻、决策点 | v0.9 |
| Deep Research | 深度研究、问题展开 | v0.9 |
| Translation | 转写翻译、摘要翻译 | v1.1 |
| Podcast | 播客化脚本或音频 | v1.1 |

## 4. MiniMax M3 与本地 ASR 分工方案

### 4.1 明确结论

截至 2026-06-12，从 MiniMax 官方文档可以确认：

1. `MiniMax-M3` 是 M 系列最新语言模型，支持 1,000,000 token 上下文。
2. `MiniMax-M3` 面向 agentic reasoning、tool use、coding、long-context tasks。
3. OpenAI-compatible Chat Completions 文档明确支持 text、image、video 输入。
4. MiniMax 官方 Audio/Speech 文档主要描述 TTS、voice clone、voice design 等能力。
5. 官方文档没有明确把 `MiniMax-M3` 标为 ASR/STT 语音转写接口。

因此，工程上不能把“MiniMax M3 直接做语音转写”作为主线承诺。ASR 主线改为本地 WhisperKit / Argmax OSS，MiniMax M3 只负责转写后的语义理解。

### 4.2 推荐主线

```text
音频文件
-> AsrProvider
-> LocalWhisperKitProvider
-> transcript_segments
-> SpeakerProvider / LocalSpeakerKitProvider
-> speaker_segments
-> TranscriptRevisionService / MiniMax M3
-> corrected_transcript_segments
-> LocalCorrectionMemory
-> WorkspaceBuilder
-> MiniMax M3
-> recording_type / summary / meeting_minutes / todo / mind_map / research
```

主线分工：

| 模块 | 推荐 provider | 职责 |
| --- | --- | --- |
| 语音转写 | WhisperKit / Argmax OSS | 本地音频转文字、时间戳、基础分段 |
| 说话人分离 | SpeakerKit / Pyannote-based diarization | 本地识别 speaker label 与时间范围 |
| 语义理解 | MiniMax M3 | 转写修正、类型识别、摘要、纪要、Todo、脑图、研究、跨会话整合 |
| TTS/播客 | MiniMax Speech 系列或后续 provider | 后置版本能力 |

### 4.3 本地 WhisperKit / SpeakerKit 方案

WhisperKit 已并入 Argmax Open-Source SDK。该 SDK 当前包含：

1. `WhisperKit`：本地 speech-to-text。
2. `SpeakerKit`：本地 speaker diarization，说话人分离。
3. `TTSKit`：本地 text-to-speech，后置版本再评估。

第一阶段不建议直接做 Swift / Tauri 深度桥接。更稳的接入方式是先把 `argmax-cli` 当作本地 sidecar 或 local server：

```text
Tauri Rust
-> 启动 / 管理 argmax-cli serve
-> 调用 http://127.0.0.1:<port>/v1/audio/transcriptions
-> 标准化为 AsrOutput
-> 写入 transcript_segments
```

这样可以复用 OpenAI Audio API 风格的接口，降低第一版集成成本。后续如果性能、稳定性和包体都满足要求，再考虑直接把 Argmax OSS Swift Package 接进 macOS 原生层。

### 4.4 说话人分离边界

后续人声分离可行，但这里的“分离”指 speaker diarization：

```text
Speaker 1 在 00:01-00:05 说话
Speaker 2 在 00:06-00:12 说话
```

它不等于 source separation，不承诺把多人同时说话的混合音频拆成多条干净人声音轨。

产品承诺应写成：

1. 支持自动识别不同说话人。
2. 支持 `Speaker 1 / Speaker 2 / Speaker 3` 标注。
3. 支持用户手动重命名、合并、修正说话人。
4. 不承诺重叠说话场景下 100% 准确。
5. 不承诺自动识别真实姓名，除非后续增加声纹注册和本地 speaker identification。

### 4.5 MiniMax M3 实验线

可以保留一个实验 provider：

```text
MinimaxM3AsrProbe
```

用途：

1. 用 feature flag 隐藏，不进入默认路径。
2. 只对短音频样本做验证。
3. 如果 MiniMax 后续正式支持音频输入或 ASR endpoint，再升级为正式 provider。
4. 评估指标包括 WER、时间戳质量、延迟、成本、失败率。

验收条件：

| 指标 | 目标 |
| --- | --- |
| 短音频转写 | 能稳定返回可解析文本 |
| 时间戳 | 至少能映射到句子级，否则不能替代 ASR |
| 说话人 | 至少能输出 speaker label，否则仍需独立 diarization |
| 稳定性 | 连续 20 次样本调用无结构化解析失败 |
| 成本 | 单小时音频成本可接受并可在设置页展示 |

## 5. 版本迭代方案

### v0.4 架构止血与本地 ASR 路线切换

目标：停止继续堆叠旧架构，为“本地 ASR + MiniMax M3 语义理解”路线打地基。

交付内容：

1. 归档旧 PRD、旧技术设计、旧 v2.0 设计包。
2. 新增 provider 分层：
   - `AsrProvider`
   - `SpeakerProvider`
   - `SemanticProvider`
   - `EmbeddingProvider`，预留
   - `ExportProvider`，预留
3. 新增 `LocalWhisperKitProvider` 的接口设计，第一阶段通过 `argmax-cli` sidecar / local server 接入。
4. 新增 `LocalSpeakerKitProvider` 的接口设计，先以 speaker diarization 为目标。
5. 新增 MiniMax M3 语义 provider。
6. 删除旧本地 LLM Todo 运行时，不再保留 Qwen / llama.cpp 回滚入口。
7. 新增模型调用记录表 `model_invocations`。
8. 新增统一 AI 产物表 `semantic_artifacts`。
9. 拆分 `src-tauri/src/lib.rs`：
   - `commands/`
   - `app/`
   - `domain/`
   - `infra/`
   - `providers/`
   - `jobs/`

验收标准：

1. `npm run build` 通过。
2. `cargo check --manifest-path src-tauri/Cargo.toml` 通过。
3. ASR provider、Speaker provider 和 Semantic provider 可独立配置。
4. Todo 提取只进入 MiniMax M3 语义产物边界，不再依赖旧本地 LLM 或旧直连云端 Todo 路径。
5. 本地 ASR 与云端 MiniMax M3 的隐私边界在设置页可见。

### v0.5 转写与说话人分离评估版

目标：用 WhisperKit 跑通本地转写，用 SpeakerKit 跑通本地说话人分离评估，对齐“语音识别与说话人分离测试评估”的核心场景。

交付内容：

1. 支持导入本地音频文件。
2. 支持录音切片任务化。
3. 接入 `argmax-cli` local server 或 CLI：
   - `POST /v1/audio/transcriptions`
   - 本地模型选择：`tiny`、`base`、`small`、`medium`、`large-v3-turbo`
   - 输出格式标准化为 `AsrOutput`
4. 新增本地模型管理：
   - 首次下载进度
   - 模型缓存目录
   - 离线可用状态
   - 设备性能检测与模型推荐
5. 新增转写任务状态机：
   - `queued`
   - `running`
   - `succeeded`
   - `failed`
   - `retrying`
6. 新增 `transcript_segments` 标准结构：
   - `session_id`
   - `audio_segment_id`
   - `speaker_id`
   - `start_ms`
   - `end_ms`
   - `text`
   - `confidence`
   - `provider`
7. 接入 `LocalSpeakerKitProvider`：
   - 输出 speaker 时间段
   - 合并到转写片段
   - 支持 `Speaker 1 / Speaker 2` 标注
8. 新增说话人表与重命名能力。
9. 新增转写评估页：
   - 音频播放
   - 分段转写
   - 说话人标签
   - 时间跳转
   - 错误片段标注

验收标准：

1. 一段音频能生成时间轴转写稿。
2. 点击任意转写片段能跳到对应音频时间。
3. 说话人 label 可重命名并持久化。
4. 失败任务可以重试。
5. 断网状态下可完成本地转写。
6. 说话人分离错误可由用户手动修正。

### v0.6 转写修正与类型化纪要工作台

目标：把本地 ASR 产出的原始转写稿修正为可用于语义处理的可信文稿，并根据会议或录音类型生成合适的摘要、纪要和待办候选。

交付内容：

1. 新增 `TranscriptRevisionService`，以原始 `transcript_segments`、speaker 信息、时间戳和本地修正记忆为输入，调用 MiniMax M3 生成修正文稿。
2. 修正范围限定为可追溯的文本质量问题：
   - ASR 错识别的专有名词、人名、产品名、缩写。
   - 断句、标点、语气词、重复词导致的可读性问题。
   - 低置信度片段中的疑似漏字、漏词、语义断裂。
   - speaker label 与上下文明显不一致的片段提示。
3. 新增转写修正对照能力：
   - 保留原始文本。
   - 生成修正文稿版本。
   - 标记改动级别：`none`、`punctuation`、`wording`、`meaning_affecting`。
   - 每处修正保留来源 `segment_id` 和修改理由摘要。
4. 新增本地修正记忆 `transcript_correction_patterns`：
   - 存储“常见错识别 -> 推荐修正”的规则摘要。
   - 存储领域词、人名、项目名、speaker alias 等本地上下文。
   - 不存完整转写稿，不存完整音频路径，不把隐私文本写入日志。
   - 用户可查看、禁用、删除；低置信度规则不得自动静默套用。
5. 新增 `WorkspaceBuilder`，只把修正文稿、必要来源索引和用户允许的修正记忆发送给 MiniMax M3。
6. 新增录音类型识别：
   - `meeting`
   - `interview`
   - `lecture`
   - `brainstorming`
   - `customer_call`
   - `personal_note`
   - `other`
7. 根据录音类型选择结构化输出模板：
   - 会议：结论、议题、决策、分歧、风险、待办。
   - 访谈：问题、回答、观点、证据、可引用片段。
   - 课程或讲座：主题、知识点、例子、术语、复习问题。
   - 个人录音：想法、情绪/动机、下一步、待整理问题。
8. 生成并存储：
   - `semantic_artifacts(type='transcript_revision')`
   - `semantic_artifacts(type='recording_type')`
   - `semantic_artifacts(type='summary')`
   - `semantic_artifacts(type='meeting_minutes')`
   - `semantic_artifacts(type='todo_extraction')`
9. 前端新增修正文稿与结构化纪要入口：
   - 原文 / 修正文稿对照。
   - 修正原因和来源片段。
   - 摘要、纪要、待办候选分区展示。
   - 手动重新修正、重新生成、拒绝某条修正。

验收标准：

1. 同一会话可看到原始转写稿和修正文稿，且能定位到对应音频时间段。
2. 摘要、纪要、待办候选默认使用修正文稿，不直接消费原始 ASR 文本。
3. M3 输出包含录音类型、模板 ID、摘要、纪要和待办候选。
4. 任何影响语义的修正都必须保留来源片段和修改理由摘要。
5. 本地修正记忆可查看、可删除、可禁用，且不存完整隐私文本。
6. M3 返回 JSON 解析失败时有可见错误和重试入口。

### v0.7 待办中枢

目标：让 Todo 从 v0.6 的“待办候选”升级为“可管理行动项”。

交付内容：

1. 新增 `todo_candidates`，承接 v0.6 的 `semantic_artifacts(type='todo_extraction')`。
2. M3 输出或重生成待办候选：
   - title
   - detail
   - owner
   - due_at
   - priority
   - confidence
   - source_spans
3. 用户确认后进入 `todos`。
4. 新增去重与合并逻辑。
5. 新增 Todo 来源追溯面板。
6. Todo 支持状态：
   - `open`
   - `in_progress`
   - `done`
   - `dismissed`

验收标准：

1. 每条 Todo 都能定位到原始转写片段。
2. 用户可以接受、编辑、忽略候选 Todo。
3. 重复 Todo 不会静默生成多条正式任务。

### v0.8 思维脑图

目标：增加用户明确要求的“生成思维脑图”能力。

交付内容：

1. 新增 `MindMapView`。
2. M3 生成 mind map JSON：
   - root
   - nodes
   - edges
   - summary
   - source_spans
3. 脑图来源支持：
   - 当前修正文稿
   - 当前摘要
   - 多个会话合并
4. 支持节点编辑、折叠、重新生成。
5. 支持导出 Markdown / JSON。
6. 存储为 `semantic_artifacts(type='mind_map')`，必要时再拆独立 `mind_maps` 表。

验收标准：

1. 一段完整会话可生成可视化脑图。
2. 脑图节点可追溯到摘要或转写片段。
3. 用户修改后的脑图不会被重新生成直接覆盖。

### v0.9 Moment 与深度研究

目标：从“整理内容”升级到“发现价值”。

交付内容：

1. 新增 Moment：
   - 关键观点
   - 决策点
   - 分歧点
   - 风险信号
   - 高价值片段
2. 新增 Deep Research 草稿：
   - 研究问题
   - 背景信息
   - 待验证假设
   - 推荐检索方向
   - 可执行下一步
3. 支持从某个片段发起研究。
4. 支持把研究结论转成 Todo 或脑图节点。

验收标准：

1. 会话能自动生成 3-10 个关键 Moment。
2. 每个 Moment 有时间戳和来源片段。
3. 深度研究结果能转为行动项。

交付内容：

1. 导出 Markdown。

3. 本地分享快照。
4. 会话归档与搜索。

### v1.1 翻译与播客化

目标：增加衍生内容能力，但不阻塞主线。

交付内容：

1. 转写翻译。
2. 摘要翻译。
3. 播客脚本生成。
4. 可选 TTS 输出。
5. 多语言导出模板。

验收标准：

1. 翻译结果与原文片段对应。
2. 播客脚本不覆盖原始摘要。
3. TTS 能作为独立 provider 开关。

### 6.4 本地 ASR Sidecar 流程

第一版本地 ASR 不直接把 Swift Package 嵌入 Tauri。推荐先用 sidecar / local server：

```text
RecordView / Import
-> Tauri command
-> RecordingService / TranscriptionService
-> LocalWhisperKitProvider
-> Argmax local server or CLI
-> AsrOutput
-> transcript_segments
-> LocalSpeakerKitProvider
-> speaker_segments
-> TranscriptRevisionService
-> MiniMax M3 transcript_revision
-> corrected transcript + correction memory
-> WorkspaceBuilder
-> MiniMax M3 recording_type + semantic artifacts
```

Sidecar 管理职责：

1. 检测 `argmax-cli` 是否存在。
2. 检测模型是否已下载。
3. 启动、停止、健康检查 local server。
4. 将本地 server 的 OpenAI Audio API 风格返回值转换为项目内部 `AsrOutput`。
5. 失败时保留原始错误摘要，但不得输出完整音频路径和隐私文本。

后续如果验证稳定，再评估 Swift 原生桥接：

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| sidecar / local server | 改造小、验证快、和现有 Rust HTTP provider 模式一致 | 进程管理、端口、安装路径需要处理 |
| Swift Package 深度桥接 | macOS 原生、性能和体验上限更高 | 工程改造大，Tauri/Rust/Swift 边界复杂 |

### 7.3 `transcript_correction_patterns` 建议字段

本地错误沉淀不应理解为“保存所有错误原文”。更准确的定义是：把用户确认过或模型高置信度识别出的可复用修正规则，保存为本地预处理上下文，在后续文稿上传 MiniMax M3 前用于提示、替换建议或术语表补充。

```sql
CREATE TABLE transcript_correction_patterns (
  id TEXT PRIMARY KEY,
  pattern_type TEXT NOT NULL,
  wrong_text TEXT NOT NULL,
  corrected_text TEXT NOT NULL,
  scope TEXT NOT NULL,
  confidence REAL NOT NULL,
  source_revision_id TEXT,
  enabled INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

`pattern_type` 建议枚举：

```text
asr_misrecognition
domain_term
person_name
project_name
speaker_alias
punctuation
segmentation
```

约束：

1. 默认只保存短语级规则，不保存完整会议转写。
2. `scope` 必须区分全局、项目、会话或 speaker，避免错误规则污染所有后续文稿。
3. 只有用户确认或高置信度规则可进入上传前预处理。
4. 每次套用规则必须能在修正对照中被用户回看和撤销。

## 8. Provider 契约

### 8.1 ASR Provider

```rust
trait AsrProvider {
    async fn transcribe(&self, input: AsrInput) -> Result<AsrOutput, AppError>;
}
```

`AsrOutput` 必须包含：

1. 原始 provider 响应摘要。
2. 分段文本。
3. `start_ms` / `end_ms`。
4. 可选 speaker label。
5. 可选 confidence。

首期实现：

| Provider | 形态 | 说明 |
| --- | --- | --- |
| `LocalWhisperKitProvider` | 本地 sidecar / local server | 默认 ASR 主线 |
| `MinimaxM3AsrProbe` | 实验 feature flag | 只用于验证，不进入默认路径 |

### 8.2 Speaker Provider

```rust
trait SpeakerProvider {
    async fn diarize(&self, input: SpeakerInput) -> Result<SpeakerOutput, AppError>;
}
```

`SpeakerOutput` 必须包含：

1. `speaker_id` 或临时 speaker label。
2. `start_ms` / `end_ms`。
3. 可选置信度。
4. 可选 `exclusive` 标记，用于解决同一时间段只归属一个 speaker 的 UI 展示。

首期实现：

| Provider | 形态 | 说明 |
| --- | --- | --- |
| `LocalSpeakerKitProvider` | 本地 sidecar / Swift SDK 后置 | 只承诺 diarization，不承诺音轨级 source separation |

### 8.3 Semantic Provider

```rust
trait SemanticProvider {
    async fn generate_artifact(
        &self,
        artifact_type: ArtifactType,
        input: SemanticInput,
    ) -> Result<SemanticArtifactDraft, AppError>;
}
```

MiniMax M3 首期支持：

1. `transcript_revision`
2. `recording_type`
3. `summary`

## 10. UI 信息架构

建议第一层导航：

| 导航 | 用户问题 |
| --- | --- |
| 录音 | 我怎么开始记录？ |
| 转写 | 刚才说了什么？谁说的？ |
| 修正 | 这段转写是否可信？哪些地方被修正？ |
| 摘要 | 这段内容的结论是什么？ |
| 待办 | 我接下来要做什么？ |
| 脑图 | 这段内容的结构是什么？ |
| 研究 | 哪些点值得继续深入？ |
| 设置 | 模型、密钥、隐私如何配置？ |

页面原则：

1. 第一屏直接进入可用工作台，不做营销式首页。
2. 录音、转写、摘要、待办、脑图之间保持同一 session 上下文。
3. 摘要、纪要、Todo、脑图都默认使用修正文稿，并能回跳到原始转写片段。
4. AI 生成内容不能静默覆盖用户编辑内容。
5. 转写修正记忆必须提供查看、禁用、删除入口。

## 11. 重构顺序

推荐顺序：

1. 先拆 provider，不先做新 UI。
2. 先接 `LocalWhisperKitProvider`，确保本地转写能跑通。
3. 再接 `LocalSpeakerKitProvider`，确保 speaker segments 能和 transcript segments 合并。
4. 再接 `TranscriptRevisionService`，确保修正文稿和原始片段可双向追溯。
5. 先建 `semantic_artifacts` 和 `transcript_correction_patterns`，再做摘要、纪要、脑图、研究。
6. 先让 Todo 可追溯，再追求 Todo 智能。
7. 先完成 v0.5 本地转写与说话人分离评估，再完成 v0.6 转写修正和类型化纪要，再做 v0.8 脑图。
7. 无老用户前提下直接删除旧本地 LLM Todo 路径，避免继续维护双路线。

不建议：

1. 不建议一次性重写整个 Tauri 后端。
2. 不建议直接把 MiniMax M3 当 ASR 主线。
3. 不建议继续把火山、阿里、腾讯云 ASR 作为默认主线；成本不符合当前产品阶段。
4. 不建议第一期做 Swift/Tauri 深度桥接；先用 `argmax-cli` sidecar 或 local server 降低集成风险。
5. 不建议把 diarization 写成“人声分离音轨”，第一期只承诺“谁在什么时候说话”。
6. 不建议每个 AI 功能单独建一套表和一套任务流程。
7. 不建议先做分享页、播客、翻译，这些不是当前瓶颈。

## 12. 风险与规避

| 风险 | 影响 | 规避 |
| --- | --- | --- |
| MiniMax M3 不能直接转写音频 | 语音主链路失败 | ASR 独立 provider，M3 只做语义主线 |
| 本地模型包体较大 | 安装包、首次下载、磁盘占用增加 | 模型按需下载，设置页展示模型大小和缓存目录 |
| 老 Mac 性能不足 | 转写延迟高，用户误以为失败 | 设备检测、模型推荐、任务进度、允许切换小模型 |
| SpeakerKit / Pyannote 许可证和模型访问条件 | 商用发布存在合规风险 | 上线前审查 MIT、CC-BY、Hugging Face 访问条款和 attribution 要求 |
| 说话人分离准确率不稳 | 转写稿可信度下降 | v0.5 先做评估页和人工修正，不承诺真实姓名识别 |
| 重叠说话无法稳定处理 | 多人同时说话片段 speaker 错配 | UI 标记低置信度，允许用户拆分、合并、重命名 speaker |
| 云模型涉及隐私 | 用户不愿上传文稿给 MiniMax M3 | 音频默认本地处理；文稿上传前给开关和提示 |
| 修正记忆污染后续文稿 | 错误规则被反复套用，导致摘要和纪要偏离原文 | 规则必须有置信度、作用域、来源和用户禁用入口；低置信度规则只作为提示，不自动替换 |
| AI JSON 解析失败 | 摘要、Todo、脑图中断 | 强 schema、重试、错误可见、保留原始文本 |
| 单体文件继续膨胀 | 后续无法维护 | v0.4 优先拆分目录和服务边界 |
| 语义模型成本不可控 | 长文稿摘要、脑图、研究调用费用高 | `model_invocations` 记录 token、耗时、估算费用；提供手动触发和自动触发开关 |
