# v0.2.0 发布说明

## 1. 发布目标

本次 `v0.2.0` 的目标是交付一版可运行的桌面端本地 Todo 提取能力，核心路线为：

- 桌面端应用：`Tauri + React`
- 本地 Todo 提取：`Qwen3-4B-Instruct-2507`
- 推理运行时：`llama.cpp`
- 量化版本：`Q4_K_M`
- Provider：`embedded_local`

本次版本中，代码、文档、运行时配置已进入 Git 仓库；真实模型权重和本地推理二进制不进入 Git 仓库，改为通过 GitHub Release 附件分发。

## 2. 版本信息

- 版本号：`0.2.0`
- Git tag：`v0.2.0`
- 主分支合并提交：`21a05a5`

## 3. Release 标题

建议使用：

```text
v0.2.0 - 内嵌本地 Qwen Todo 提取
```

## 4. Release 资产清单

### 4.1 必传资产

以下文件建议作为 GitHub Release 附件上传：

1. `qwen3-4b-instruct-2507-q4_k_m.gguf`
   - 说明：Qwen3-4B-Instruct-2507 的 GGUF 量化模型
   - 作用：本地 Todo 提取推理权重

2. `llama-cli`
   - 说明：`llama.cpp` 命令行可执行文件
   - 作用：本地运行时健康检查与推理入口

3. `llama-completion`
   - 说明：`llama.cpp` 补全文本模式可执行文件
   - 作用：当前本地 Todo 提取调用入口

4. `checksums.sha256`
   - 说明：Release 附件校验文件
   - 作用：校验下载后资产完整性

### 4.2 可选资产

1. `smart-todo-desktop-v0.2.0.dmg`
   - 若本次产出 macOS 安装包，建议同时上传

2. `smart-todo-desktop-v0.2.0.zip`
   - 若本次提供压缩包形式分发，可作为补充资产

3. `release-notes-v0.2.0.md`
   - 若需要单独保留一份纯文本发布说明，可选上传

## 5. 资产命名建议

建议统一使用以下命名，避免后续版本混淆：

- 模型文件：`qwen3-4b-instruct-2507-q4_k_m.gguf`
- 推理二进制：`llama-cli`、`llama-completion`
- 校验文件：`checksums.sha256`
- 安装包：`smart-todo-desktop-v0.2.0.dmg`

## 6. checksums.sha256 生成命令

在待上传资产所在目录执行：

```bash
shasum -a 256 qwen3-4b-instruct-2507-q4_k_m.gguf llama-cli llama-completion > checksums.sha256
```

如果同时给安装包做校验，可执行：

```bash
shasum -a 256 qwen3-4b-instruct-2507-q4_k_m.gguf llama-cli llama-completion smart-todo-desktop-v0.2.0.dmg > checksums.sha256
```

## 8. 发布前检查清单

发布 `v0.2.0` 前，至少完成以下检查：

1. 确认远端已有 `main` 最新提交
2. 确认远端已有 tag：`v0.2.0`
3. 确认 Release 附件命名与文档一致
4. 确认 `checksums.sha256` 与待上传资产一致
5. 确认安装包版本号为 `0.2.0`
6. 确认应用启动后本地运行时状态为 `ready`
7. 确认“未回退”场景文案显示为 `不适用`
8. 确认文档无乱码，且为 UTF-8（无 BOM）

## 9. 发布步骤

1. 打开 GitHub 仓库 Releases 页面
2. 选择 `v0.2.0`
3. 填写 Release 标题
4. 粘贴本文件中的 Release 正文模板
5. 上传 Release 附件
6. 校对附件名称、版本号和正文内容
7. 发布 Release

## 10. 回滚建议

若 `v0.2.0` Release 资产上传错误：

1. 先删除错误附件
2. 重新生成 `checksums.sha256`
3. 重新上传正确资产
4. 若正文描述已失真，立即同步修正 Release 正文

若已发布后发现模型资产错误：

1. 不修改既有 tag 指向
2. 先更新 Release 正文中的已知问题说明
3. 补传修正资产并注明替换时间
4. 若影响较大，追加 `v0.2.1` 修复版本
